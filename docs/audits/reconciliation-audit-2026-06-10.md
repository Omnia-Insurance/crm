### fetchOverrides reads policy number via hardcoded snapshot keys and ignores carrier scoping — learned overrides silently lost and cross-carrier contamination

- **Where:** `packages/twenty-server/src/modules/reconciliation/services/review-item.service.ts:830`
- **Severity:** high | **Lens:** architecture-generics | **Status:** confirmed (2 independent verifiers)

fetchOverrides extracts the carrier policy number from bobRowSnapshot via two hardcoded header guesses: `snapshot.policy_number ?? snapshot.Policy_Number` (lines 829-833). But bobRowSnapshot is keyed by the raw XLSX headers from the parse job, and Ambetter's canonical header is 'Policy Number' (space-separated title case — see parsers/ambetter.ts:9 and config/ambetter.field-config.ts:24; transforms.ts resolveFieldMapping exists precisely because both title-case and underscore header formats occur). For title-case files, neither key matches, the item is silently skipped, and a human's approved UNMATCHED/MULTI_MATCH decision never becomes a Tier-1 OVERRIDE on the next run — defeating the feature's purpose with zero error signal. Separately, the query (lines 810-814) filters only on decision/category/matchMethod with no carrier or reconciliation constraint, then stamps every result with the caller-passed carrierName (lines 836-841). Once a second carrier exists, approvals from carrier A are returned as overrides for carrier B, and matchRow's `o.carrierName === carrierName` check (engines/matching.ts:570) is always satisfied because the value was fabricated. Tier-1 overrides are confidence-100 AUTO_MATCHED, so this can silently mis-match policies across carriers.

**Recommendation:** Resolve the policy-number header once via the reconciliation's columnMapping (the entry whose crmField === 'policyNumber', exactly as match.job.ts:713 and parse.job.ts:123 already do) and persist a first-class `carrierPolicyNumber` column (plus `carrierName`) on the reviewItem record at creation time in match.job, so fetchOverrides becomes a typed `repo.find({ where: { carrierName, decision: 'APPROVED', ... } })` with no snapshot key guessing.

### Unknown parserVersion silently disables the status engine and turns every clean matched row into a ghost UPDATE review item

- **Where:** `packages/twenty-server/src/modules/reconciliation/engines/status.ts:391`
- **Severity:** medium | **Lens:** architecture-generics | **Status:** confirmed (2 independent verifiers)

STATUS_ENGINES (status.ts:377-379) contains only 'ambetter-bob-v1'; deriveStatus returns null for any other parserId rather than erroring (status.ts:389-393). match.job.ts:657 falls back to 'ambetter-bob-v1' when parserVersion is null, but if a new carrier sets any other parserVersion, derivedStatus stays null for every row — and deriveCategory (types/field-config.ts:126-130) computes `statusChanged = derivedStatus !== currentCrmStatus` without null-guarding derivedStatus. With derivedStatus=null and a real CRM status, statusChanged is always true, so every matched row with zero field diffs is promoted to category 'UPDATE' (match.job.ts:530-537) — flooding review with items that have no diffs, no STATUS_CHANGE flag (deriveFlags does null-guard, field-config.ts:168), and nothing to apply. The inverse failure is equally bad: leaving parserVersion null for a non-Ambetter carrier silently applies Ambetter's eligible-for-commission/payment-error semantics to foreign data.

**Recommendation:** In loadMatchContext, validate parserId against an exported `Object.keys(STATUS_ENGINES)` (or a `isKnownStatusEngine(id): id is StatusEngineId` guard) and throw a descriptive error so the run fails fast at FAILED/MATCH instead of producing garbage review items. In deriveCategory, treat statusChanged as `derivedStatus !== null && derivedStatus !== currentCrmStatus && !isNegativeToNegativeStatusChange(...)`. Remove the implicit 'ambetter-bob-v1' default and make parserVersion required on CarrierConfig.

### carrierConfig.matchingConfig cast to MatchingConfig wholesale with no validation or defaults merge — partial JSON crashes the match job

- **Where:** `packages/twenty-server/src/modules/reconciliation/jobs/match.job.ts:658`
- **Severity:** medium | **Lens:** architecture-generics | **Status:** confirmed (2 independent verifiers)

loadMatchContext does `(carrierConfig.matchingConfig as MatchingConfig) ?? DEFAULT_MATCHING_CONFIG` (match.job.ts:658-660). CarrierConfigRecord types the field as `Record<string, unknown> | null` (types/reconciliation.ts:93), and it is an admin-editable JSON field on a workspace record. The `??` only protects against null — any non-null object replaces the defaults entirely. A config saved before a field existed, or hand-edited, that lacks `enabledTiers` throws TypeError at engines/matching.ts:404 (`config.enabledTiers.includes`); lacking `tier7NameBands`/`tier7ConfidenceScores` throws at matching.ts:814-821 mid-row. The same unvalidated-cast pattern repeats for statusConfig (match.job.ts:667), fieldConfig-as-ComputedFieldDef[] (match.job.ts:671-672, parse.job.ts:89-90), and `new RegExp(carrierConfig.policyNumberPattern)` (match.job.ts:663-665) which throws on an invalid pattern. All failures surface only as an opaque FAILED reconciliation.

**Recommendation:** Add a `parseCarrierPipelineConfig(record: CarrierConfigRecord)` boundary function (suggest types/carrier-config.ts) that does `{ ...DEFAULT_MATCHING_CONFIG, ...stored }` (or zod `MatchingConfigSchema.partial()` + defaults) and validates the regex/computed-field shapes with actionable error messages. Type CarrierConfigRecord.matchingConfig as `Partial<MatchingConfig> | null` so the compiler forces the merge at every consumer.

The broker-effective audit heuristic exists twice with different inputs and predicates. ### State machine is not compare-and-swap: concurrent triggers can duplicate review items, and setFailed lets stale job retries stomp healthy runs to FAILED

- **Where:** `packages/twenty-server/src/modules/reconciliation/services/state-machine.service.ts:32`
- **Severity:** medium | **Lens:** architecture-generics | **Status:** confirmed (2 independent verifiers)

transition() validates the transition against the caller-supplied currentStatus, then writes via an unconditional `repo.update({ id }, data)` (mutation.service.ts:29) — there is no `WHERE status = :expected` guard. The orchestrator does read-then-transition (orchestrator.service.ts:25-35), so two near-simultaneous startReconciliationMatching calls (double-click, or a queue retry racing a manual restart via FAILED→MATCHING) both pass validation and enqueue two match jobs. persistMatchResults' idempotency (delete-then-create, match.job.ts:388-399) is per-job, not cross-job: batchCreate sleeps 500ms between 200-row batches (review-item.service.ts:136-148), giving job B ample window to delete mid-create or interleave creates → duplicated/partial review item sets. Separately, setFailed deliberately bypasses transition validation (state-machine.service.ts:56-74), so a retried parse job that re-runs after a successful chain (its PARSING→PARSED transition now throws because status is REVIEW) force-writes the whole reconciliation to FAILED while its review items are live.

**Recommendation:** Make transition a true CAS: `repo.update({ id, status: currentStatus }, data)` and throw/no-op when affected === 0; have both jobs re-read the reconciliation at start and exit cleanly if status isn't the expected PARSING/MATCHING. In setFailed, skip the write when the current status is already a post-step state (REVIEW/COMPLETED) reached by a newer run.

- **Severity:** high | **Lens:** parsers-config | **Status:** confirmed (hand-verified after verifier outage)

Three corruption paths in the shared date transform that every date-typed column flows through: (1) transforms.ts:82-90 accepts any `\d{1,2}/\d{1,2}/\d{4}` without range-checking month/day, so a DD/MM-formatted carrier file ('31/12/2025') yields the invalid ISO string '2025-31-12' (and days <= 12 silently swap month/day). ### Auto-approve effect fires server APPLY (with rule-learning cascade) on mere item selection, and retries forever on error

- **Where:** `packages/twenty-front/src/modules/reconciliation/components/MatchedDiffView.tsx:674`
- **Severity:** high | **Lens:** frontend | **Status:** confirmed (hand-verified after verifier outage)

The useEffect at MatchedDiffView.tsx:674-700 promotes PENDING -> APPROVED whenever `allDiffsAccepted` is true. `allDiffsAccepted` (:634-667) compares the persisted fieldDiffs' bobValue against the *live* record-store value, so it is true whenever the CRM record already equals the BOB values — not only after the user inline-accepts. **Recommendation:** Make transitions compare-and-swap: UPDATE ... SET status = :target WHERE id = :id AND status = :current and treat affected=0 as a conflict (job aborts). Pass a per-run token/jobId in the payload and have persistMatchResults verify it before delete/create; use the driver's options.id dedup when enqueuing.

### Worker crash mid-job leaves the reconciliation permanently stuck in PARSING/MATCHING with no recovery path

- **Where:** `packages/twenty-server/src/modules/reconciliation/services/state-machine.service.ts:6`
- **Severity:** medium | **Lens:** jobs-pipeline | **Status:** reported by auditor, not independently verified

Jobs are enqueued with no retryLimit, and the BullMQ driver defaults to attempts = 1 (bullmq.driver.ts:263), so a job is never re-delivered. If the worker process dies mid-parse or mid-match (OOM, deploy restart — plausible given both jobs hold the whole file/policy book in memory), the catch/setFailed never runs and status stays PARSING or MATCHING forever. VALID_TRANSITIONS offers no exit: PARSING only →PARSED|FAILED and MATCHING only →REVIEW|FAILED, and the orchestrator's restart paths require FAILED (FAILED→PARSING|MATCHING), so startReconciliationParsing/Matching throw 'Invalid status transition' and the run is unrecoverable without manual SQL. The same zombie occurs when setFailed's own write fails — it logs and swallows the error (state-machine.service.ts:79-84), leaving the record in the in-flight state while the job message is already consumed. There is also a narrower variant in parse.job.ts:219-229: status is set to MATCHING before queue.add, so a crash between the two leaves MATCHING with no match job ever enqueued.

**Recommendation:** Add stuck-state recovery: either allow PARSING→PARSING / MATCHING→MATCHING re-entry from the orchestrator, or record a startedAt/heartbeat and let a watchdog (or the orchestrator, when the state is older than a threshold) force FAILED. Consider a small retryLimit on enqueue, and enqueue the match job before flipping status to MATCHING.

### readSourceFile picks the newest attachment of any kind — an unrelated later attachment silently replaces the BOB source

- **Where:** `packages/twenty-server/src/modules/reconciliation/services/attachment.service.ts:72`
- **Severity:** low | **Lens:** jobs-pipeline | **Status:** reported by auditor, not independently verified

The source spreadsheet is located by raw SQL: SELECT ... FROM attachment WHERE "targetReconciliationId" = $1 ORDER BY "createdAt" DESC LIMIT 1 — no filter on name, extension, or mime type. Attachments on a reconciliation record can be added through Twenty's standard record-page attachment UI (notes, screenshots, carrier emails). **Recommendation:** Store the source attachment id (or fileId) on the reconciliation record at upload time and read exactly that file; at minimum filter the query by xlsx/csv extension or mime type and surface which attachment was used in the run stats.

### Transforms never throw, so the parse job's cell-error machinery and stats.parseErrors are unreachable — bad values silently become null and drive wrong statuses

### Leaving a comment unconditionally overwrites the decision with FLAG_AUDIT, destroying APPROVED bookkeeping and undo eligibility

### Apply all / Undo all / Reject have no error handling — failures are unhandled promise rejections with zero user feedback, and writes are duplicated client+server

## 3. Improvements: customizability, generics, architecture (20)

### Heuristically-narrowed winners inherit Tier 2/3/5 confidence (90-98) and the misleading POLICY_NUMBER_SINGLE method

- **Where:** `packages/twenty-server/src/modules/reconciliation/engines/matching.ts:719`
- **Severity:** medium | **Lens:** matching-engine | **Status:** unjudged (verifier outage)

After the narrowing pre-pass reduces N candidates to 1 (matching.ts:633-635), the surviving candidate flows through the normal tiers: a recency-based guess between two CANCELED policies that fails date/agent checks still hits Tier 5 at confidence 90 AUTO_MATCHED with method POLICY_NUMBER_SINGLE and the note 'Single CRM policy matched by policy number' (matching.ts:718-731) — even though the tier's own justification comment ('the Ambetter U-number is unique per member', :715-717) no longer holds, and only the notes suffix reveals the guess. By contrast, the genuinely-ambiguous path (Tier 6 weighted scoring) is capped at 70 and NEEDS_REVIEW (matching.ts:775). So weaker evidence (a last-resort ops heuristic) produces strictly higher confidence than the engine's own ambiguity handling. This matters downstream: batchApproveReviewItems(minConfidence) and learned decision rules treat confidence as trust, so recency-narrowed guesses at 90-98 are sweepable into bulk approval.

**Recommendation:** When narrowedWinner came from selectByMostRecentEffectiveDate (and arguably selectByActiveTerm), cap or discount the returned confidence (e.g. below autoMatchThreshold so status becomes NEEDS_REVIEW), and/or use a distinct MatchMethod (e.g. POLICY_NUMBER_NARROWED_RECENT) so review filters, batch approval, and learned rules can distinguish heuristic narrowing from a genuinely unique identifier match.

### Tier 8 DOB matching depends on exact raw-string equality, unlike the diff engine's tolerant Date comparison

- **Where:** `packages/twenty-server/src/modules/reconciliation/engines/matching.ts:848`
- **Severity:** low | **Lens:** matching-engine | **Status:** unjudged (verifier outage)

Tier 8 looks up candidates via policyByDob.get(input.memberDob) and re-checks `leadDob === input.memberDob` (matching.ts:843-848); Tier 6 awards its +10 DOB bonus on the same strict equality (:765). The BOB side is normalized to 'YYYY-MM-DD' by toDateString (transforms.ts:59-119), but the CRM side is whatever the ORM returns for lead.dateOfBirth, stored raw as the Map key (matching.ts:438-445, data.service.ts:155 just casts). The diff engine deliberately does NOT trust string equality for this same field — detectMultiMemberSubscriberMismatch parses both sides with new Date().getTime() (diff.ts:368-373) — an inconsistency suggesting the formats are not guaranteed identical. If lead.dateOfBirth ever carries a time component (e.g. '1990-05-15T00:00:00.000Z' from a DATE_TIME column) or any non-canonical format, Tier 8 silently never fires and the Tier 6 DOB bonus never applies: pure false negatives with no error. No test covers a CRM DOB that differs only in format.

**Recommendation:** Normalize DOB to 'YYYY-MM-DD' on the CRM side when building policyByDob (e.g. slice ISO strings / format Date values in fetchPoliciesForMatching), and compare normalized values in Tiers 6/8 — matching the timestamp-equality semantics the diff engine already uses.

### Columns mapped to CRM fields outside the hardcoded snapshot are silently never compared

- **Where:** `packages/twenty-server/src/modules/reconciliation/engines/diff.ts:790`
- **Severity:** medium | **Lens:** diff-engine | **Status:** unjudged (verifier outage)

The column-mapping loop skips any entry whose crmField is not a key of the crmPolicy snapshot (diff.ts:790: `if (!(entry.crmField in crmPolicy)) continue;`). The snapshot is a fixed, hand-maintained set of ~17 dot-paths in fetchPoliciesForMatching (data.service.ts:143-164) plus 4 enrichment fields (data.service.ts:211-218). The import dialog lets users map BOB columns to arbitrary policy/lead fields, but anything outside that hardcoded list (e.g. lead.addressCustom.addressCity, planName, lead.jobTitle) is silently dropped — no diff, no warning, no INFO_ONLY entry — so users reasonably believe a mapped column is reconciled when it is a no-op. This is the central trap of the 'columnMapping drives the diff' design: the comparable field set is actually defined in data.service.ts, and the two can drift without any signal (the match job already logs a warning for a similar drift on matching-shaped columns, match.job.ts:155-159, but the diff path has nothing).

**Recommendation:** Either derive the fetched/enriched field set from the reconciliation's columnMapping (fetch what is mapped), or log a per-run warning and surface an INFO_ONLY diff/flag listing mapped-but-uncomparable crmFields so config drift is visible to reviewers.

### Currency handling is contradictory: unreachable toMicrosString conversion in the column loop, and no .amountMicros suppression or conversion in the computed-field loop

- **Where:** `packages/twenty-server/src/modules/reconciliation/engines/diff.ts:803`

Commit cd1028b48e suppresses all '.amountMicros' columnMapping entries at diff.ts:777 (`continue`), which makes the dollars→micros conversion from the earlier d24fdcec74 commit at diff.ts:803-805 (and toMicrosString itself, diff.ts:610-615) unreachable dead code in that loop. ### State machine validates against caller-supplied status, and APPLYING/COMPLETED states are unreachable dead code

- **Where:** `packages/twenty-server/src/modules/reconciliation/services/state-machine.service.ts:32`
- **Severity:** medium | **Lens:** status-and-rules | **Status:** unjudged (verifier outage)

`transition()` validates the move using the `currentStatus` argument the caller passes in; it never reads the stored status and performs no locked/conditional update (state-machine.service.ts:25-49). Both jobs hardcode their claimed current status: parse.job.ts passes 'PARSING'→'PARSED' then 'PARSED'→'MATCHING' (parse.job.ts:192-224), match.job.ts passes 'MATCHING'→'REVIEW' (match.job.ts:408-411). The guard is therefore advisory only: a queue-retried match job that previously setFailed will overwrite FAILED with REVIEW; a duplicate/stale match job running concurrently with a user-restarted one (FAILED→MATCHING) both pass validation and both run delete-all-review-items + batchCreate + auto-rule CRM writes (match.job.ts:388-406), risking duplicated review items and clobbered decisions. Separately, the APPLYING and COMPLETED states (VALID_TRANSITIONS lines 11-13) are unreachable: since the 2026-04-29 removal of the server-side apply step (orchestrator.service.ts:73-77), no server code and no frontend code (grep of twenty-front/src/modules/reconciliation finds zero APPLYING/COMPLETED references) ever transitions into them, so every reconciliation terminates at REVIEW forever and appliedAt/completedAt are never set.

**Recommendation:** Have transition() read the stored status inside a conditional UPDATE (UPDATE ... SET status = target WHERE id = :id AND status = :expected, checking affected rows) so validation reflects reality and concurrent/retried jobs fail fast instead of clobbering state. Then either delete the dead APPLYING/COMPLETED states and timestamps, or add a real terminal transition (e.g., REVIEW→COMPLETED when all review items are decided) so runs can actually complete.

### Batch apply and learned-rule auto-apply issue N+1 sequential queries with non-transactional multi-record writes per item

- **Where:** `packages/twenty-server/src/modules/reconciliation/services/review-item.service.ts:536`
- **Severity:** medium | **Lens:** jobs-pipeline | **Status:** endorsed by feasibility judge | effort: medium

applyOneReviewItem runs per item: a query-builder policy fetch with lead join (:536-540), then up to four sequential UPDATEs (policy :569, person :576, cancel-previous policy :787, review item :600). batchApply (:257-273) and applyLearnedRulesForReconciliation (:410-450) call it in a sequential for-loop with no batching, so a batch approve over a few thousand pending items is ~5 round trips each, executed inside a single GraphQL mutation (resolver) or inside the match job itself (called from persistMatchResults, match.job.ts:401) — long wall-clock times and request-timeout exposure as carriers grow. ### Whole-file / whole-book in-memory processing with no size guard; null carrierId silently matches against every policy in the workspace

- **Where:** `packages/twenty-server/src/modules/reconciliation/services/data.service.ts:117`
- **Severity:** low | **Lens:** jobs-pipeline | **Status:** endorsed by feasibility judge | effort: small

The pipeline holds everything in memory simultaneously: parse loads the full xlsx Buffer (parse.job.ts:74), the full raw-row array (:80), a second normalized copy (:120-169), and JSON.stringify of the whole dataset (attachment.service.ts:130); match re-reads and JSON.parses the entire parsed file (attachment.service.ts:146-152) plus every CRM policy with lead+agent relations into one array (data.service.ts:110-173). ### Silent fallback in status-role header resolution: a misnamed column yields all-null status inputs and blanket PAYMENT_ERROR derivations with zero warnings

resolveFieldMapping's fallback (transforms.ts:303-308) keeps the configured header verbatim when it matches nothing in the file — necessary for computed output keys like 'True Effective Date', but it also swallows genuine misconfigurations: if a carrier file renames 'Paid Through Date' to 'Paid Thru', the role resolves to a key absent from every row, buildStatusInputFromMapping (status.ts:98-100) reads undefined -> null, and the Ambetter engine derives PAYMENT_ERROR_ACTIVE_APPROVED for the whole book (status.ts:315-322) — wrong results, no log line anywhere in parse.job (:101-117) or match.job (:686-693).
